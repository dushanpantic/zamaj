# Plan: Exercise Library Seeding (clear-and-reseed)

**Created**: 2026-06-06
**Branch**: master
**Status**: approved

## Goal

Replace the current exercise library with a curated canonical catalog (common + specialized movements) that ships embedded in the app. A one-time destructive migration clears the existing `LibraryExercises` table (the app is unreleased, single-install — matching the existing v7 `_wipeAllDomainTables` precedent and its "no compat layer" rationale), then the canonical catalog seeds idempotently on launch. Each exercise captures `prominence` (common/specialized, used as a sort key) and primary/secondary muscle data (captured now; the muscle filter UI is a deferred follow-up). UUIDs are generated by a committed tool script (never hand-authored) and an append-only frozen-id test guarantees idempotency can never silently break.

**Nomenclature:** the prominence tiers are `common` and `specialized` (chosen over mainstream/niche — neutral register, no dismissive tone, states the sort intent: common sorts first).

Because the library is emptied before seeding, there are no duplicates to reconcile — this collapses what was a 3-phase feature into one. The previously-scoped `LibraryMerger` service and reconcile UI are **dropped**. After the wipe, the user re-links existing program exercises to canonical entries via the *existing* exercise-editor library picker, choosing "Keep local" to preserve their own names (e.g. "BB Bench" stays "BB Bench" while linked to "Barbell Bench Press"). Full design rationale: `memory/project_exercise_library_seeding.md`.

## Acceptance Criteria

- [ ] The v11→v12 migration clears all `LibraryExercises` rows and sets every program exercise's `libraryExerciseId` to NULL; programs, exercise names, sets, and all session history are otherwise untouched.
- [ ] On launch, the library is populated with the embedded canonical catalog (common + specialized).
- [ ] Seeding is idempotent: re-running it (subsequent launches, app updates) never duplicates an entry and never overwrites user edits to a previously-seeded row.
- [ ] The library list and the in-session picker order exercises common-first, then specialized, alphabetical within each tier.
- [ ] Every exercise carries `prominence`, `primaryMuscles`, and `secondaryMuscles`; primary and secondary muscle sets are disjoint.
- [ ] Seed UUIDs are produced by `tool/assign_seed_uuids.dart` (real v4s), never written by hand; the tool only fills entries missing an id and never regenerates an existing one.
- [ ] A frozen-id test fails if any previously-shipped seed id is changed or removed from the catalog (ids are append-only).
- [ ] A catalog integrity test enforces: unique 36-char UUIDv4 ids, trimmed non-empty names, valid `measurementType`/`prominence`/`MuscleGroup` enum values, non-empty `primaryMuscles`, disjoint primary/secondary muscles.
- [ ] `tool/ci.sh` passes (offline-imports guard, codegen, format, analyze, test).

## User-Facing Behavior

No `/specs` artifacts exist for this task; the following scenarios are authored from the locked decisions and drive the TDD steps.

```gherkin
Feature: Pre-populated exercise library (clear-and-reseed)

  Scenario: Upgrade wipes the old library and seeds the canonical catalog
    Given an existing install with manually-created library entries
    And programs whose exercises link to some of those entries
    When the app upgrades to the seeded schema and launches
    Then the library contains exactly the canonical catalog entries
    And the old manual library entries are gone
    And the programs and their exercise names are unchanged
    And those program exercises are now unlinked (libraryExerciseId is null)
    And all session history is unchanged

  Scenario: Entries are ordered common-first
    Given the catalog has been seeded
    When the user opens the library list or the in-session picker
    Then entries appear common-first, then specialized, alphabetical within each tier

  Scenario: Re-seeding is idempotent and preserves user edits
    Given the catalog has already been seeded
    And the user has renamed a seeded entry and added a video URL to it
    When the app launches again
    Then no duplicate rows are created
    And the user's rename and video URL are preserved

  Scenario: Re-linking preserves the local exercise name (existing behavior)
    Given a program exercise named "BB Bench"
    When the user links it to the canonical "Barbell Bench Press" and chooses "Keep local"
    Then the exercise stays named "BB Bench" in the program
    And its libraryExerciseId points to "Barbell Bench Press"

  Scenario: Muscle data is captured for every exercise
    Given a seeded canonical exercise
    Then it exposes a non-empty primary muscle list
    And its primary and secondary muscle sets do not overlap

  Scenario: Existing-id integrity is enforced at build time
    Given the catalog has shipped with a set of seed ids
    When a developer changes or removes an existing entry's id
    Then the frozen-id test fails
```

## Risks & Open Questions

- **Destructive wipe is one-time and intentional.** It runs only in the `from < 12` migration block, so it fires once on upgrade and never again. Consistent with the v7 `_wipeAllDomainTables` precedent for this single-install app. After it, the user re-links program exercises via the existing picker, choosing "Keep local" to preserve names.
- **Domain version stays at 8 (not bumped).** New fields are first-class columns with safe defaults; no existing serialized payload shape changes. *Confirm with architecture before Step 4*; if the convention is to bump on any model-shape change, bump 8 → 9 and regenerate goldens.
- **Prominence ordering must be explicit, not alphabetical.** Step 5 uses an explicit rank (common before specialized) so a future tier rename can't silently reorder the list.
- **rootBundle layering.** Asset IO lives only in `bootstrap.dart`; the seeder service takes a JSON string to stay pure and guard-compliant.
- **Catalog content quality.** I author a standard catalog (~60–80 movements); the user will prompt later to add anything missing. Muscle tags and measurement types (e.g. pull-ups = bodyweight, plank = timeBased) should get a human eyeball before merge.
- **Future progress-tracking design note.** Session snapshots freeze `libraryExerciseId` immutably, so the wipe orphans the ids in past snapshots. When the progress feature is built, correlate exercise history via the **live template link** (`plannedExerciseIdInSnapshot` → live `Exercises` row → its current `libraryExerciseId`), not the frozen snapshot id — that way re-linking unlocks full retroactive history (for sessions whose program exercise still exists). Do not rewrite past snapshots (immutability invariant).

## Plan Review Summary

Reviewed inline across four lenses (acceptance/test, design, UX, strategic). Verdict: **approve with observations** — no blockers.

**Acceptance / test**
- Resolved: seed-failure behavior is now specified (log-and-continue; Step 7) so a bad asset can't brick launch.
- Resolved: non-empty `primaryMuscles` is enforced by the catalog integrity test (Step 6), not the model invariant (user-created entries may have none).

**Design / architecture**
- The destructive wipe is the simplification that removes the merge service and reconcile UI — net structural reduction.
- Muscles stored as a JSON-array column (not a join table); fine at this scale, but the eventual muscle filter will be a JSON/LIKE scan, not an indexed join — note when planning the filter.
- `LibrarySource` survives the phase cut as informational metadata (distinguish canonical-seed rows from later user additions); cheap to keep.

**UX**
- The picker/list jumps to ~80+ entries; search (`nameQuery`) must stay prominent; common-first ordering mitigates scroll load.
- Post-wipe, program exercises are unlinked until the user re-links them — a one-time chore; document it in `product-context.md` so it doesn't read as data loss.

**Strategic**
- Clear-and-reseed turns a 3-phase feature into one cohesive slice with no orphaned intermediate states.
- Catalog authoring (~80 movements, correct muscle tags) is the dominant content effort; the integrity + frozen-id tests backstop correctness, and the user will iterate on coverage.
