# Exercise Library — Implementation Plan

## Goal

Give every distinct movement (e.g. "BB Bench Press") a single durable identity, so that template `Exercise` rows on different workout days (PUSH/UPPER) and substitute exercises chosen mid-session can all point at the same library entry. This unlocks cross-day/cross-program progress aggregation in a later release.

## Design decisions (locked in discussion)

1. **New global entity `LibraryExercise`** lives in its own module (`lib/modules/exercise_library/`). One row per movement, ever. Global — not scoped to a program.
2. **Template `Exercise` gains a nullable `libraryExerciseId`.** Existing `name`, `measurementType`, `metadata.{notes,videoUrl}`, `plannedRestSeconds` stay on `Exercise` — they remain the snapshot-captured truth so the "completed sessions never change retroactively" rule keeps holding for free.
3. **`SubstituteExercise` also gains a nullable `libraryExerciseId`.** In-session swaps participate in the join from day one. (Substitute payloads are persisted as canonical JSON inside `session_exercises.substitute_payload_json`, so the snapshot rule still holds — the libraryExerciseId is frozen at swap time.)
4. **Linking is optional.** Existing rows ship with `libraryExerciseId = null`. The user opts in either per-row (Exercise card chip) or in bulk (Library screen → "Suggest from your programs").
5. **Soft-delete via `archivedAt`.** Archived entries disappear from the picker but still resolve for past data, so the join from old sessions is never dangling.
6. **Out of scope for v1:** progress charts. v1 only collects the linkage and ships the management UI. Charts come in a follow-up against real data.

## Layer / module placement

- Domain: `lib/modules/domain/models/library_exercise.dart` + `lib/modules/domain/repositories/exercise_library_repository.dart`.
- Persistence: new Drift table in `tables.dart`, new repo impl in `lib/modules/persistence/repositories/drift_exercise_library_repository.dart`, mapper in `lib/modules/persistence/mappers/`.
- UI: new module `lib/modules/exercise_library/` (screens, widgets, bloc, navigation). UI talks to data only via `ExerciseLibraryRepository` — never imports drift directly (enforced by `tool/check_offline_imports.sh`).
- Cross-module exports: barrel files `domain.dart`, `persistence.dart`, `exercise_library.dart`.

## Schema-version bumps

`lib/core/schema_versions.dart`:
- `drift`: 10 → 11.
- `domain`: 7 → 8.

Both go up together because the change adds a new table **and** new fields on persisted domain entities (`Exercise.libraryExerciseId`, `SubstituteExercise.libraryExerciseId` inside the substitute payload).

## Implementation order

Build bottom-up — domain first, then persistence + migration, then repo wiring, then UI. Don't start a layer until the layer below is green (analyze + tests).

### Step 3 — Drift schema + migration

**Edit:** `lib/modules/persistence/database/tables.dart`

Add a new table:

```dart
@TableIndex(name: 'library_exercises_name_lower', columns: {#nameLower})
class LibraryExercises extends Table {
  TextColumn get id => text().withLength(min: 36, max: 36)();
  TextColumn get name => text()();
  TextColumn get nameLower => text()();  // trim+lowercase, for picker search + collision check
  TextColumn get measurementTypeDiscriminator => text()();
  TextColumn get measurementTypePayloadJson => text()();
  TextColumn get videoUrl => text().nullable()();
  TextColumn get cues => text().nullable()();
  IntColumn  get archivedAtMs => integer().nullable()();
  IntColumn  get createdAtMs => integer()();
  IntColumn  get updatedAtMs => integer()();
  IntColumn  get schemaVersion => integer()();

  @override Set<Column> get primaryKey => {id};

  // No DB-level UNIQUE on nameLower — collision detection is a UX prompt,
  // not an invariant (users can deliberately archive + re-create).
}
```

Add a new column to `Exercises`:

```dart
TextColumn get libraryExerciseId => text()
    .nullable()
    .references(LibraryExercises, #id, onDelete: KeyAction.setNull)();
```

`onDelete: setNull` matters because hard-deletes are technically possible (e.g. via a future "purge archived" tool) — we don't want to cascade and destroy template data.

`substitute_payload_json` on `SessionExercises` stays unchanged at the column level; the new field rides inside the JSON payload.

### Step 5 — Repository contract

**New file:** `lib/modules/domain/repositories/exercise_library_repository.dart`

```dart
abstract class ExerciseLibraryRepository {
  Future<LibraryExercise> create({
    required String name,
    required MeasurementType measurementType,
    String? videoUrl,
    String? cues,
  });

  Future<LibraryExercise?> get(String id);

  /// Returns active (non-archived) entries by default. Pass
  /// [includeArchived: true] for management screens.
  Future<List<LibraryExercise>> list({
    bool includeArchived = false,
    MeasurementType? measurementType,
    String? nameQuery,           // case-insensitive substring; null = no filter
  });

  Future<LibraryExercise> update(LibraryExercise entry);

  Future<LibraryExercise> archive(String id);
  Future<LibraryExercise> unarchive(String id);

  /// Case-insensitive trimmed lookup. Returns the matching entry if one exists
  /// (archived or active), null otherwise. Used by the collision check on
  /// "Add to library."
  Future<LibraryExercise?> findByNormalizedName(String name);
}
```

Add to `domain.dart` barrel export.

## What's *not* in this plan (deferred)

- **Progress charts / history aggregation.** Once linked data starts flowing, the chart UI gets designed against real data.
- **Seeded starter catalog.** No baked-in list of common movements. Users build their library as they go (or via the suggestion screen).
- **Muscle group / equipment / category metadata.** Deferred until the filter actually earns its keep.
- **Library across devices / sync.** App is offline-first; sync is out of scope for the whole project today.
- **Hard-delete UX.** Archive only. A "purge archived" tool can come later if archived clutter becomes a problem.
- **Renaming-with-history surfacing.** If a library entry's name diverges from a linked template row's `name`, v1 just shows the library name in the chip. No "row labelled 'Bench'" annotation for now.

## Risks / things to watch

- **Picker performance.** With a few hundred library entries, the picker is fine in-memory. If users build huge catalogs, add SQL-side `nameLower LIKE ?` filtering before sorting in Dart.
- **Schema-version coupling.** Bumping `domain` to 8 means every persisted row gets re-stamped to 8 on next write; that's expected and matches the existing pattern, but worth being deliberate.
- **`measurementType` immutability on library entries.** v1 locks it from creation. If a user mis-types it, they must archive + recreate. Acceptable trade-off; revisit if it becomes painful.
- **Substitute payload backwards-compat.** The new `libraryExerciseId` key is optional and absent in pre-v8 payloads. The freezed `SubstituteExercise.fromJson` must default it to `null` (it will, because the factory parameter is nullable with no default — `json_serializable` reads missing keys as null for nullable types). Confirm in the serialization test.
